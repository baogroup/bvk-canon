BVK CANON
Модель публикации архива на сайте

Статус: ACTIVE
Версия: v1.0
Дата первого принятия: 2026-04-26
Текущая редакция: 2026-04-26
История поправок: GitHub History
Уровень доступа: SERVICE
Историческая принадлежность: служебный архивный слой / слой сайта и публикации
Связанные закрытые документы: нет
Коды связанных закрытых документов: нет

Назначение:
Этот файл фиксирует рабочую модель публикации архивных документов BVK на сайте, чтобы не превращать WordPress или иной публичный сайт в ручную копию тысяч файлов.

Главный принцип:
GitHub остаётся источником истины для канона.
Сайт остаётся читательским слоем, навигацией и публичным входом в архив.
PDF используется только для избранных документов, сборников и важных редакций, а не как основной способ хранения каждого файла.

## 1. Почему нельзя вручную копировать всё на сайт

Если документов будут сотни и тысячи, ручное создание отдельной публикации для каждого файла на сайте создаст несколько проблем:

- дублирование источников;
- риск, что версия на сайте устареет;
- огромная ручная нагрузка;
- сложность обновления;
- путаница между каноном и его копией;
- тяжёлое обслуживание ссылок;
- невозможность быстро проверить историю документа.

Поэтому сайт не должен становиться вторым ручным архивом.

## 2. Рекомендуемая трёхслойная модель

### Слой 1. GitHub как источник истины

В GitHub хранится:

- CURRENT;
- PUBLIC_READ;
- SERVICE-документы;
- история версий через GitHub History;
- структура папок;
- индексы;
- связи;
- исходные текстовые файлы;
- будущие SVG, изображения, PDF и сборки.

GitHub отвечает на вопрос:
где находится действующий канон и его история?

### Слой 2. GitHub Pages / assistant-public слой как читаемый архив

Из GitHub можно собирать читаемый HTML-слой:

- публичный архив;
- assistant-слой;
- HTML-версии документов;
- индексные страницы;
- страницы по годам;
- страницы по типам документов;
- страницы по темам.

Этот слой отвечает на вопрос:
как удобно читать документы без входа в технический GitHub-интерфейс?

### Слой 3. Основной сайт BVK.lt как публичный вход

На bvk.lt должны быть не тысячи ручных копий документов, а:

- главная навигация;
- новости;
- объяснения проекта;
- страницы рубрик;
- избранные документы;
- ссылки на публичный архив;
- карточки важных документов;
- тематические подборки;
- публикационные страницы, которые ведут к архивным основаниям.

Сайт отвечает на вопрос:
как читатель входит в мир и понимает, куда идти дальше?

## 3. Как ссылаться на документы

Для обычного читателя лучше ссылаться не на github.com, а на читаемые HTML-страницы GitHub Pages или публичного архива.

Допустимые варианты ссылок:

1. Внутренняя страница bvk.lt с краткой карточкой документа и ссылкой на архив.
2. Страница GitHub Pages с красиво собранным HTML-документом.
3. Прямая ссылка на GitHub только как „source / canonical file“ для тех, кто хочет видеть исходник.
4. PDF только для важных документов или сборников.

## 4. Роль PDF

PDF не должен быть основным архивным форматом для каждого документа.

PDF подходит для:

- конституционных сборников;
- больших законов;
- годовых архивных сборок;
- публичных брошюр;
- версий для скачивания;
- важных редакций;
- печатных материалов;
- избранных исторических документов.

Для тысяч мелких записей PDF создаст лишнюю нагрузку.

## 5. Как лучше делать архив на сайте

На bvk.lt стоит создать раздел:

Archyvas

Внутри него:

- 2026 metų archyvas;
- Pirmųjų metų kronikos;
- Atviri klausimai;
- Dokumentai;
- Chronologija;
- Ateities naujienų atramos;
- Viešos archyvinės ištraukos;
- Svarbiausi dokumentai.

Каждая страница сайта может быть не полной копией всех документов, а навигационной карточкой:

- название документа;
- дата;
- код;
- тип;
- краткое описание;
- уровень доступа;
- ссылка „Skaityti archyve“;
- ссылка „Canonical source“ на GitHub при необходимости.

## 6. Новости и архивные ссылки

Если новость ссылается на первичный документ, в конце новости можно добавить блок:

Susiję archyvo dokumentai:
- A-2026-01-27-01 — Pirmųjų galimų produktų sąrašas
- FC-2026-10 — Pirmiau archyvas, o tik po to naujienos

Ссылки должны вести на читаемый публичный архив, а не обязательно на техническую страницу GitHub.

## 7. Что делать на раннем этапе

Пока документов немного, допустимо временно делать проще:

1. Документ живёт в GitHub.
2. GitHub Pages / assistant слой показывает читаемую HTML-версию.
3. На bvk.lt создаётся страница архива с короткими карточками и ссылками.
4. Только самые важные документы можно дополнительно публиковать как отдельные страницы сайта.

Не нужно сразу переносить всё вручную в WordPress.

## 8. Что делать на позднем этапе

Когда документов станет много, нужен автоматический или полуавтоматический механизм:

- скрипт сборки HTML из CURRENT/PUBLIC_READ;
- генерация индексов по годам;
- генерация карточек документов;
- возможно импорт в WordPress custom post type;
- автоматическое обновление ссылок;
- отдельные страницы для PUBLIC-документов;
- скрытие SERVICE, RESTRICTED и SEALED из публичного сайта.

## 9. Рекомендуемое решение

Краткая модель:

GitHub = канон и история.
GitHub Pages = читаемый архив.
bvk.lt = вход, новости, объяснения, карточки и ссылки.
PDF = избранные документы и сборники.
WordPress-публикации = только важные публичные тексты, а не каждый файл архива.

## 10. Формула

Не копировать архив вручную.
Не делать PDF для всего.
Не ссылаться обычному читателю только на сырой GitHub.

Собирать читаемый архив из GitHub и вести к нему с сайта.

Вывод:
архив BVK должен быть технически устойчивым и читательски понятным. GitHub хранит правду канона, сайт открывает к ней дверь.
