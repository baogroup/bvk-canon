BVK CANON
Правило датирования документов по слою времени

Статус: ACTIVE
Версия: v1.0
Дата первого принятия: 2026-04-26
Текущая редакция: 2026-04-26
История поправок: GitHub History
Уровень доступа: SERVICE
Историческая принадлежность: служебный слой архива и датировки документов
Связанные закрытые документы: нет
Коды связанных закрытых документов: нет

Назначение:
Этот документ уточняет, какие даты должны стоять в шапках документов BVK Canon, если документ относится не к реальному времени создания файла, а к внутренней хронологии проекта или будущей Республики.

Главный принцип:
Pirmojo priėmimo data и Dabartinė redakcija должны относиться к дате документа внутри канона, а не к реальной технической дате создания файла в GitHub.

GitHub History хранит реальную техническую историю создания, правки и коммитов.

## 1. Архивные документы внутри хронологии

Если документ оформлен как архивная запись 2027, 2035, 2046, 2086 или любого другого года внутренней хронологии, его поля должны использовать дату этого документа в мире канона.

Пример:

Pirmojo priėmimo data: 2027-03-09
Dabartinė redakcija: 2027-03-09
Pakeitimų istorija: GitHub History

Даже если файл технически создан в GitHub в 2026 году, в шапке архивного документа должна стоять дата внутреннего документа.

## 2. Служебные документы текущей работы

Если документ является настоящим рабочим SERVICE-документом, созданным для управления каноном, правилами, генератором, структурой или ассистентским слоем, он может использовать реальную дату принятия правила.

Пример:

Дата первого принятия: 2026-04-26
Текущая редакция: 2026-04-26
История поправок: GitHub History

## 3. Индексы годовых архивов

Годовые индексы могут иметь дату первого создания внутри соответствующего архивного года, если они оформлены как часть внутреннего архива.

Если индекс является служебным рабочим инструментом, допустимо оставить реальную дату его создания, но это должно быть ясно из уровня доступа и исторической принадлежности.

## 4. Что хранит GitHub History

GitHub History хранит:

- реальную дату создания файла;
- автора коммита;
- все изменения;
- техническую историю правок;
- сравнение версий.

Поэтому не нужно смешивать техническую дату GitHub с датой документа внутри мира.

## 5. Практическое правило

Архивная запись будущего года:
дата в шапке = дата события / документа внутри канона.

Рабочее правило канона:
дата в шапке = реальная дата принятия рабочего правила.

История поправок:
всегда GitHub History, если нет отдельной крупной канонической версии.

## 6. Формула

Дата документа — внутри мира.
История файла — в GitHub.

Вывод:
Это правило нужно, чтобы будущие архивные документы выглядели как документы своего времени, а GitHub оставался технической памятью реального процесса создания канона.
