BVK CANON
Правило области применения CHANGELOG

Статус: ACTIVE
Версия: v1.0
Дата первого принятия: 2026-04-26
Текущая редакция: 2026-04-26
История поправок: GitHub History; CHANGELOG применяется только для крупных канонических изменений
Уровень доступа: SERVICE
Историческая принадлежность: служебный архивный слой
Связанные закрытые документы: нет
Коды связанных закрытых документов: нет

Назначение:
Этот файл уточняет роль CHANGELOG в BVK Canon и защищает проект от избыточной бюрократии при росте архива до сотен и тысяч документов.

Главный принцип:
GitHub History является полной технической историей всех изменений файлов.
CHANGELOG используется не для каждой правки и не для каждого нового документа, а только для крупных канонических, структурных и правовых изменений.

## 1. Почему нужен ограниченный CHANGELOG

BVK Canon должен вырасти в большой архив: записи основателя, очерки, протоколы, проекты решений, законы, новости, годовые планы, документы экономики, медицины, безопасности, производства, образования, культуры и инфраструктуры.

Если каждое добавление такого документа фиксировать отдельной записью CHANGE, журнал станет тяжёлым, медленным и неудобным.
Это создаст бюрократическую нагрузку, которая не добавляет доказательности, потому что техническая история уже хранится в GitHub.

## 2. Что фиксирует GitHub History

GitHub History фиксирует:

- каждое создание файла;
- каждую правку;
- автора правки;
- дату правки;
- предыдущие версии;
- сравнение изменений;
- историю коммитов.

Для обычных документов этого достаточно.

## 3. Что фиксирует CHANGELOG

CHANGELOG фиксирует только крупные изменения:

- изменение структуры канона;
- создание нового крупного раздела;
- изменение правил доступа PUBLIC / SERVICE / RESTRICTED / SEALED;
- изменение основной хронологии 2026-2086;
- изменение статуса BVK, Республики, 2046 или 2086 года;
- изменение конституционного или правового каркаса;
- изменение правил новостей и документальной опоры;
- изменение архивной политики;
- переименование, отмену или перенос важного канонического документа;
- создание крупной версии или редакционного рубежа;
- изменение правил работы ассистента с каноном.

## 4. Что не нужно фиксировать в CHANGELOG

В CHANGELOG не нужно вносить:

- каждую новую архивную запись;
- каждую новость;
- каждый рабочий очерк;
- каждый протокол обсуждения;
- каждую заметку основателя;
- каждую правку текста;
- исправление ошибок;
- добавление обычного документа в годовой архив;
- обновление SEO-блока;
- правку изображения или промта;
- техническое уточнение, не меняющее канон.

Такие изменения фиксируются через GitHub History и, при необходимости, через годовые индексы.

## 5. Годовые индексы вместо бесконечного CHANGELOG

Для больших архивных массивов используются годовые индексы:

- CURRENT/01_ARCHIVE/2026/00_ARCHIVE_INDEX_2026.txt
- CURRENT/01_ARCHIVE/2027/00_ARCHIVE_INDEX_2027.txt
- CURRENT/01_ARCHIVE/2028/00_ARCHIVE_INDEX_2028.txt

Годовой индекс показывает, какие документы существуют в рамках года, их даты, коды, типы и краткое назначение.

## 6. Практическое правило для ассистента

При создании обычных документов ассистент должен:

1. Проверить, не существует ли файл уже.
2. Создать или обновить документ.
3. Использовать понятный commit message.
4. При необходимости обновить годовой индекс.
5. Не добавлять запись в CHANGELOG, если изменение не является крупным каноническим поворотом.

При создании крупного правила, нового раздела или изменении архитектуры канона ассистент должен предложить отдельную запись в CHANGELOG или подготовить её для ручного внесения.

## 7. Формула

GitHub History хранит всё.
Годовые индексы помогают ориентироваться.
CHANGELOG фиксирует только большие повороты канона.

Вывод:
CHANGELOG остаётся важным инструментом, но перестаёт быть бухгалтерией каждого файла. Он становится журналом крупных решений, а не тормозом живого архива.
