Назначение:
Этот файл закрепляет простой порядок редактирования канона без потери связности и без создания дубликатов.

Как работать:
1. Сначала открыть assistant-слой и свериться, существует ли нужный файл.
2. Если файл уже существует, редактировать его, а не создавать новый дубликат.
3. Открыть нужный файл в CURRENT.
4. Исправить текст.
5. При необходимости обновить шапку файла:
- Версия
- Текущая редакция
- История поправок
- Уровень доступа
- Историческая принадлежность
- Связанные закрытые документы
- Коды связанных закрытых документов
6. Если документ уровня PUBLIC связан с RESTRICTED или SEALED, обязательно отметить это в шапке через поля закрытых связей и их коды.
7. Добавить запись в CHANGELOG.txt.
8. Сохранить изменения через Commit changes.
9. После коммита запустить Build and deploy BVK site.
10. Проверить обновлённую страницу в assistant-слое и при необходимости в PUBLIC_READ.

Что считается источником истины:
- действующий текст: CURRENT
- история мелких и обычных правок: GitHub History
- отдельные крупные сохранённые редакции: VERSIONS
- отменённые документы: REPEALED

Что нельзя делать:
- создавать новый файл вслепую, не сверившись с assistant-слоем
- забывать обновлять CHANGELOG.txt
- оставлять PUBLIC-документ со скрытой закрытой связью без отметки в шапке
- считать assistant-страницу обновлённой до запуска сборки сайта

Вывод:
Сначала проверка существующих файлов, потом правка CURRENT, потом запись в CHANGELOG, потом сборка сайта и контроль результата.
